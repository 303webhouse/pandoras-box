**Ship this first. It is live and dangerous right now.** Verified `hub_get_flow_radar` at `ca68c01` returns `net_premium_direction:"BULLISH"` on `calls $0 / puts $0 / 0 events`, `status:"ok"`. Given Nick's bull-bias / post-windfall-sizing failure modes, fake-bullish confirmation is the worst possible direction to be wrong in.

**Root cause** — `backend/api/flow_radar.py :: _compute_flow_radar()`, §5 Market pulse:
`overall_pc` falls back to `0` when `total_call == 0`, and `0 < 0.7` → `"BULLISH"`. Two bugs in one line: (1) empty feed → BULLISH; (2) all-puts/zero-calls real flow (`total_call==0, total_put>0`) hits the same `0` fallback → reads BULLISH on genuinely bearish flow.

## CHUNK 2 — Excise the dead `_flow_aligned` v2-classifier flow arm  · INDEPENDENT · AFTER-HOURS-SAFE

**ATLAS decision: EXCISE, do not rename-repair.** Here is why the obvious "just fix the keys" is a trap:

- **Key mismatch (real):** `scoring/feed_tier_classifier_v2.py` reads `net_call_premium`/`net_put_premium`/`net_premium`; the only writer (P4A, `signals/pipeline.py:476–481`) produces `total_premium`/`call_premium`/`put_premium`. Every read falls back to `0` → `_flow_aligned` is permanently `False` → `flow_ok` can never be True → the `fully_confirmed` badge (pythia AND flow AND sector) is structurally impossible. Dead 2 months.
- **Scope mismatch (the deeper trap):** `net_call_premium`/`net_premium` are **market-tide** fields (market-wide, read by `committee_bridge.py`). The classifier looked for *tide-shaped* data in a *per-ticker* flow dict. The `$500K` `FLOW_NET_PREMIUM_THRESHOLD` is calibrated for tide-scale numbers. So a naive key-rename leaves the arm **inert** anyway (per-ticker `call−put` rarely clears a tide-scale threshold) — i.e. repair-in-name-only: a NEW fake-healthy. Reject.
- **Why excise is safe:** the v2 classifier is shadow-only (`FEED_TIER_USE_V2=false`, not authoritative). The flow arm is already dead and `fully_confirmed` already unreachable, so **excising changes zero current behavior.** L1a rebuilds flow confluence properly on the canonical `flow_events` source with a per-ticker-calibrated threshold. One-line conceptual reversal: L1a re-introduces it correctly.
