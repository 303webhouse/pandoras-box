## Task 2 — Real `staleness_seconds` (server-computed, None-safe) · ATLAS
Compute **server-side** from `data_updated_at` so there's one clock and one threshold definition. **Preferred:** factor the compute into a single shared helper both paths call (guarantees they can't diverge); inlining the identical formula in both is acceptable.

```python
from datetime import datetime, timezone
def _staleness_from(dua):  # dua: ISO str or None
    if not dua:
        return None
    return int(max(0, (datetime.now(timezone.utc) - datetime.fromisoformat(dua)).total_seconds()))
```

**(a) MCP envelope** — `backend/hub_mcp/tools/flow_radar.py` (~L119-141): read `market_pulse.get("data_updated_at")`, compute `staleness`, pass to `make_response(..., staleness_seconds=staleness)`. **Replaces the hardcoded `300`.**

**(b) Dashboard HTTP response** — wherever the dashboard flow endpoint serializes `market_pulse` (located in Phase 0): emit the same server-computed `staleness_seconds` (int ≥0 or `null`) alongside `data_updated_at`, identical formula.

**Invariant (P0):** `staleness_seconds` is `null` when age is unknown — **NEVER 0**. Zero reads as "perfectly fresh" = a fake-healthy lie.

## Task 3 — Dead-feed watchdog alarm + recovery (RTH-gated, debounced) · ATLAS
A **cadence-driven** check. **NEVER fire from the request-driven read path** — it would spam on every committee call.

**Placement (Phase 0 picks the exact host; the invariant binds):** the check must catch BOTH (a) the poller writing nothing usable, and (b) **the poller stopped entirely**. ⚠️ **Do NOT bury the check inside `run_flow_poller`** — a dead poller would then never fire it (dead-man's-switch gap). Use a small independent monitor task, or hook an existing scheduled health loop, that evaluates `uw:flow` freshness on a cadence.

Logic each cycle:
- **Not RTH** (market-hours helper) → do nothing. Stale-after-close is correct; the `false` at 4:50pm today is expected, not an incident.
- **RTH + feed dead** (`flow_data_available` False — no fresh `uw:flow:{ticker}` keys within the 900s window):
  - If no alarm latched → `send_alert("🚨 Flow feed dead", <status>, severity="warning")` + latch a Redis flag (e.g. `alarm:flow_dead:active`, TTL ~2h) so it fires **ONCE per episode**, not every cycle.
- **RTH + feed healthy again + latch set:**
  - `send_alert("✅ Flow feed restored", <status>, severity="info")` + clear the latch.

**Stale threshold = 900s** (matches the 15-min TTL governing `flow_data_available` — widget + availability flag can never contradict).

**Alarm body hygiene (AEGIS-flavored):** `<status>` = feed facts only — ticker count, oldest `staleness_seconds`, last-write time. **NO env values, NO DSN, NO tokens.**
